# Plan 4.1: Firestore Data Layer

## Objective

Create the query and real-time subscription layer for reading tracked emails and their open/click events from Firestore. This is the foundation that the dashboard UI will consume.

## Context

- src/api/db.js ‚Äî Existing Firestore setup (app init, `logEmailSent`)
- src/firebase-config.js ‚Äî Firebase config
- src/content.js ‚Äî Auth flow (`getAuthToken`, `authenticateFirebase`) that we'll reuse
- functions/index.js ‚Äî Cloud Functions write events to `emails/{emailId}/events/{eventId}`
- firestore.rules ‚Äî Security rules require `userId == auth.uid`
- .gsd/DECISIONS.md ‚Äî Data model: emails use UUID as doc ID, events are sub-collections

## Data Model Reference

```
emails/{emailId}          ‚Üí { userId, subject, recipients: [{email, id}], sentAt }
emails/{emailId}/events/  ‚Üí { type: "open"|"click", recipientId, timestamp, userAgent, ip, targetUrl? }
```

<task type="auto">
  <name>Update Firestore security rules for read queries</name>
  <files>firestore.rules</files>
  <action>
    The current read rule uses `resource.data.userId == request.auth.uid`, which is correct for single-doc reads. However, `list` queries (used by `onSnapshot` with `where`) need a separate rule in Firestore.

    Verify the current rules support list queries. The `allow read` rule already covers both `get` and `list`, and the `where("userId", "==", userId)` in the query satisfies the security rule evaluation. No change should be needed, but verify by reviewing the rules.

    Also add a Firestore composite index if needed. Firestore will auto-create indexes for simple queries, but the `where("userId") + orderBy("sentAt")` combination requires a composite index. Create `firestore.indexes.json` with:

    ```json
    {
      "indexes": [
        {
          "collectionGroup": "emails",
          "queryScope": "COLLECTION",
          "fields": [
            { "fieldPath": "userId", "order": "ASCENDING" },
            { "fieldPath": "sentAt", "order": "DESCENDING" }
          ]
        }
      ]
    }
    ```

    Update `firebase.json` to reference this index file by adding `"indexes": "firestore.indexes.json"` under the `"firestore"` key.

  </action>
  <verify>firebase deploy --only firestore ‚Äî should deploy rules and indexes without errors</verify>
  <done>
    - firestore.indexes.json exists with the composite index
    - firebase.json references the index file
    - Deploy succeeds
  </done>
</task>
