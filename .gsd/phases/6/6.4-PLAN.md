# Plan 6.4: Config + Cutover — Rules, Indexes, Deploy & Verify

## Objective

Update Firestore security rules and indexes to match the new data model,
deploy everything, then perform the cutover (wipe old test data, verify
fresh end-to-end tracking with new paths).

## Context
- `firestore.rules` — Current rules match `emails/{emailId}`
- `firestore.indexes.json` — Has composite index on top-level `emails` collection
- `.gsd/DECISIONS.md` — Phase 6 cutover decisions

## Tasks

<task type="auto">
  <name>Update firestore.rules and firestore.indexes.json</name>
  <files>firestore.rules, firestore.indexes.json</files>
  <action>
    **firestore.rules — full replacement:**

    ```
    rules_version = '2';

    service cloud.firestore {
      match /databases/{database}/documents {

        // Per-user email collection
        match /users/{userId}/emails/{emailId} {
          allow read, write: if request.auth != null && request.auth.uid == userId;

          // Events written by Cloud Functions (Admin SDK bypasses rules).
          // Readable by the owning user only (userId from path).
          match /events/{eventId} {
            allow read: if request.auth != null && request.auth.uid == userId;
          }
        }

        // Lookup collection: written by client at send time, read by Cloud Functions.
        // Cloud Functions use Admin SDK (bypasses rules) so only client writes need a rule.
        // No read rule needed from client side — emailLookup is internal-only.
        match /emailLookup/{emailId} {
          allow create: if request.auth != null;
        }
      }
    }
    ```

    Key improvements vs old rules:
    - `userId` comes from the path param (not `resource.data.userId`) — cleaner and more correct
    - Events readable by the owning user (not just "any authed user") — tightened security
    - `emailLookup` allows client creates only; no read from client (Admin SDK reads it in Cloud Functions)

    **firestore.indexes.json — remove composite index:**

    Replace contents with:
    ```json
    {
      "indexes": [],
      "fieldOverrides": []
    }
    ```

    The old `(userId ASC, sentAt DESC)` composite index on `emails` is no longer
    needed — the new query is on `users/{uid}/emails` ordered by `sentAt` only
    (single-field, auto-indexed by Firestore).
  </action>
  <verify>
    cat firestore.rules | grep "users/{userId}/emails"
    cat firestore.indexes.json | grep -c "fieldPath"  # should output 0
  </verify>
  <done>
    - firestore.rules matches users/{userId}/emails/{emailId} path
    - events match block uses userId path param (not resource.data)
    - emailLookup match block present
    - firestore.indexes.json has empty indexes array
  </done>
</task>

       ```bash

       ```

    2. **Deploy Firestore rules + indexes** (new path structure):

       - Open Firebase Console → Firestore → `emails` collection
       - Delete the entire `emails` collection (all docs + subcollections)

    5. **Reload extension in Chrome**:

    6. **Verify end-to-end** (see success criteria below)
