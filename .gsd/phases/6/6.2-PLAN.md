# Plan 6.2: Cloud Functions — emailLookup-Based Path Resolution

## Objective

Update both Cloud Functions (`trackPixel`, `trackClick`) in `functions/index.js`
to resolve the correct Firestore path via the `emailLookup` collection instead of
expecting `userId` as a query param. Events are written to
`users/{userId}/emails/{emailId}/events` after a single lookup read.

## Context
- `.gsd/DECISIONS.md` — Phase 6 decisions (lookup collection approach)
- `functions/index.js` — File being modified
- Current tracking URLs: `?emailId=X&recipientId=Y` (no userId — lookup handles it)

## Tasks

<task type="auto">
  <name>Update trackPixel to use emailLookup for path resolution</name>
  <files>functions/index.js</files>
  <action>
    In the `trackPixel` function:

    1. After validating `emailId` and `recipientId` are present, add a lookup step:
       ```js
       const lookupSnap = await db.collection("emailLookup").doc(emailId).get();
       if (!lookupSnap.exists) {
         // Unknown emailId — still return the GIF (don't break recipient's email client)
         // just skip logging
         return respondWithGif(res);
       }
       const userId = lookupSnap.data().userId;
       ```

    2. Change the Firestore write from:
       ```js
       db.collection("emails").doc(emailId).collection("events").add(...)
       ```
       to:
       ```js
       db.collection("users").doc(userId).collection("emails").doc(emailId).collection("events").add(...)
       ```

    3. Extract the GIF response into a small inline helper to avoid duplication:
       the function must still return a 1x1 GIF even when lookup fails.

    The `userId` query param (which does NOT exist in current URLs) must NOT be
    expected — the whole point is the URL stays clean.
  </action>
  <verify>Check functions/index.js references "emailLookup" and "users" collections</verify>
  <done>
    - trackPixel reads emailLookup/{emailId} to get userId
    - trackPixel writes event to users/{userId}/emails/{emailId}/events
    - GIF still returned even if lookup fails (graceful degradation)
  </done>
</task>

<task type="auto">
  <name>Update trackClick to use emailLookup for path resolution</name>
  <files>functions/index.js</files>
  <action>
    Apply the same emailLookup pattern to `trackClick`:

    1. After validating `emailId`, `recipientId`, and `targetUrl`, add lookup:
       ```js
       const lookupSnap = await db.collection("emailLookup").doc(emailId).get();
       if (lookupSnap.exists) {
         const userId = lookupSnap.data().userId;
         await db.collection("users").doc(userId)
           .collection("emails").doc(emailId)
           .collection("events").add({ ... });
       }
       // Always redirect regardless of lookup outcome
       res.redirect(targetUrl);
       ```

    2. Change the Firestore write path to:
       `users/{userId}/emails/{emailId}/events`

    3. The redirect to `targetUrl` must happen in ALL cases — even if lookup
       fails — so the recipient is never stranded.

    Keep the existing error catch block. The lookup is inside the existing try/catch.
  </action>
  <verify>Check functions/index.js — trackClick references "users" collection path</verify>
  <done>
    - trackClick reads emailLookup/{emailId} to get userId
    - trackClick writes event to users/{userId}/emails/{emailId}/events
    - Redirect still happens even if lookup doc doesn't exist
  </done>
</task>
